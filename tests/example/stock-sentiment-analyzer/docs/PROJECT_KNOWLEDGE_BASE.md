# 股票舆情分析系统 项目知识库
## 项目概览
### 基本信息
- **项目名称**: 股票舆情分析系统 (stock-sentiment-analyzer)
- **项目类型**: AI驱动的金融分析工具
- **核心功能**: 实时采集财经资讯，多维度专家评估，生成短线交易建议
- **技术栈**: Python 3.12+, YAML配置, Markdown报告输出
### 系统价值
本系统通过自动化采集和AI分析，帮助投资者快速获取市场热点、评估舆情影响、生成交易决策参考，减少信息差带来的投资风险。

## 核心模块说明
### 1. 资讯采集模块 (`src/collectors/`)
#### 功能定位
负责从多个权威数据源实时采集财经资讯、公告、市场动态。
#### 现有实现
| 采集器 | 功能描述 | 状态 |
|--------|----------|------|
| `xueqiu_collector.py` | 雪球平台资讯采集 | ✅ 完成 |
| `eastmoney_collector.py` | 东方财富网资讯采集 | ✅ 完成 |
| `news_collector.py` | 综合新闻源采集 | ✅ 完成 |
| `cls_collector.py` | 财联社资讯采集 | ✅ 完成 |
#### 核心特性
- 7x24小时不间断采集
- 多数据源并行抓取
- 智能去重和内容过滤
- 异常重试和降级机制
### 2. 专家评估模块 (`src/experts/`)
#### 功能定位
模拟不同领域金融专家视角，对采集到的资讯进行多维度评估。
#### 专家角色配置
| 专家类型 | 权重 | 评估维度 |
|----------|------|----------|
| 宏观经济专家 | 25% | 政策影响、宏观经济走势、行业周期 |
| 行业研究员 | 25% | 行业景气度、产业链影响、公司基本面 |
| 技术面分析专家 | 20% | K线形态、量价关系、技术指标 |
| 资金面分析专家 | 20% | 资金流向、主力动向、市场情绪 |
| 风险管控专家 | 10% | 风险预警、利空因素、止损建议 |
#### 核心逻辑
- 每位专家独立输出评估结果
- 包含影响等级、影响方向、影响时长三个核心指标
- 支持自定义专家权重和评估维度
### 3. 结果汇总模块 (`src/aggregator/`)
#### 功能定位
整合多位专家的评估结果，计算加权综合得分，生成统一的评估结论。
#### 核心算法
1. 对各专家的评估结果进行标准化处理
2. 根据预设权重进行加权求和
3. 计算综合影响等级和概率分布
4. 识别高度共识和存在分歧的评估点
### 4. 交易决策模块 (`src/trader/`)
#### 功能定位
基于综合评估结果和行情数据，生成具体的短线交易建议。
#### 决策输出
- 操作建议：买入/卖出/观望
- 上涨概率：百分比值
- 目标价区间：预期收益范围
- 止损价：风险控制阈值
- 持仓周期：建议持有时长
### 5. 报告生成模块 (`src/reporter/`)
#### 功能定位
将分析结果格式化为易读的报告，支持多种输出格式。
#### 输出格式
- Markdown格式分析报告（默认）
- 通达信软件兼容格式
- 结构化JSON数据
- 文本摘要格式
### 6. 行情数据模块 (`src/data/`)
#### 功能定位
统一的行情数据接口层，屏蔽不同数据源的差异。
#### 数据源支持
| 数据源 | 类型 | 支持级别 |
|--------|------|----------|
| 通达信本地数据 | 离线 | ✅ 完全支持（日/周/月/分钟级别） |
| 东方财富API | 实时 | ✅ 支持 |
| 综合行情接口 | 实时 | 🔄 开发中 |
#### 核心特性
- 智能降级策略：API失败时自动切换到本地数据
- 多层缓存机制：减少重复请求
- 统一数据格式：上层模块无需关注数据源差异
### 7. 定时调度模块 (`src/scheduler/`)
#### 功能定位
支持定时任务执行，实现无人值守的持续分析。
#### 运行模式
| 模式 | 说明 | 使用场景 |
|------|------|----------|
| 单次模式 | 执行一次分析后退出 | 手动分析、测试 |
| 定时模式 | 按预设间隔循环执行 | 实盘监控 |
| 交互模式 | 命令行交互操作 | 调试、单次查询 |

## 重构专项知识
### 一、现有架构痛点与技术债务
#### 1. 核心耦合问题
- **主类过度耦合**：[`StockSentimentAnalyzer`](src/main.py:26)类直接初始化所有模块实例，违反单一职责原则，模块间没有抽象层，难以单独替换和测试
- **采集器硬编码**：主类中直接实例化4个采集器对象，新增/移除数据源需要修改主类代码，不符合开闭原则
- **资讯去重逻辑放在主类**：去重逻辑属于采集层职责，当前放在主流程中导致职责混乱
- **模块间没有依赖注入**：所有模块都在主类中直接实例化，无法灵活替换实现，单元测试需要启动完整系统
#### 2. 采集器模块共性问题（所有采集器都存在）
- **重复代码严重**：4个采集器（Xueqiu/EastMoney/Cls/News）都重复实现了会话管理、请求重试、超时控制、日志记录逻辑，代码复用率低于30%
- **缺少统一基类**：没有抽象公共采集器基类，每个采集器都独立实现collect()接口，接口规范不统一
- **配置解析重复**：每个采集器都单独从config字典中解析自己的配置，没有统一的配置验证机制
- **错误处理不一致**：不同采集器的异常处理策略不同，有的返回空列表，有的抛出异常
#### 3. 专家评估模块问题
- **提示词硬编码**：专家提示词直接写在代码中，无法动态调整和版本管理
- **评估逻辑耦合**：5个专家的评估逻辑耦合在同一个类中，新增/修改专家类型需要修改核心代码
- **AI调用逻辑分散**：专家模块、交易模块都有独立的AI调用逻辑，没有统一的AI调用网关
#### 4. 数据层问题
- **接口不统一**：不同数据提供器的方法命名、参数、返回值格式不统一，上层模块需要适配不同接口
- **缓存逻辑耦合**：缓存逻辑直接写在数据提供器实现中，无法灵活替换缓存策略
- **没有数据校验层**：从数据源获取的数据没有统一的校验和清洗逻辑，脏数据会直接传递到上层
#### 5. 代码结构问题
- 所有模块都直接接收完整config字典，缺乏明确的配置契约，配置变更容易导致模块运行异常
- 异步方法缺乏统一的超时和熔断机制，单个模块故障会导致整个分析流程失败
- 模块间数据传递采用无类型字典，缺乏数据结构定义，开发过程中容易出现字段缺失和类型错误
- 没有统一的错误码和异常体系，错误处理散落在各个模块中
#### 6. 性能瓶颈点
- 资讯采集为串行执行，4个采集器总耗时等于各采集器耗时之和，可并行化优化
- 专家评估采用单条资讯串行评估，可批量并行处理
- 没有增量分析能力，每次运行都需要完整执行全流程

### 四、接口规范与契约
#### 1. 采集器统一接口
所有采集器必须实现：
```python
async def collect(self) -> List[dict]:
    """采集资讯，返回资讯对象列表"""
```
#### 2. 专家评估接口
```python
async def evaluate(self, news: dict) -> List[dict]:
    """评估单条资讯，返回各专家评估结果列表"""
```
#### 3. 结果汇总接口
```python
async def aggregate(self, news: dict, expert_results: List[dict]) -> dict:
    """汇总专家评估结果，返回综合评估结果"""
```
#### 4. 交易决策接口
```python
async def decide(self, news: dict, aggregated_result: dict) -> dict:
    """生成交易决策，返回决策对象"""
```
#### 5. 报告生成接口
```python
async def generate(self, **kwargs) -> dict:
