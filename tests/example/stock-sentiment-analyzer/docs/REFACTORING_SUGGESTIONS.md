# 股票舆情分析系统 重构建议文档
## 一、整体重构策略
### 核心原则
1. **渐进式重构**：采用小步快跑的方式，每个重构点独立验证、独立上线，避免大规模集中修改
2. **向下兼容**：所有重构确保现有功能完全兼容，不改变现有输出格式和接口契约
3. **安全网优先**：重构前先补充测试用例，作为功能正确性的验证基准
4. **价值导向**：优先解决对业务影响最大、技术债务最严重的问题
### 整体路线
```
基础准备 → 模块解耦 → 功能优化 → 架构升级
    ↓         ↓         ↓         ↓
  测试用例   采集器重构  并行化改造  流程编排
  公共组件   DI容器     异常体系    AI网关
  数据模型   配置契约   可观测性    增量分析
```
## 二、优先级建议
### 🔴 高优先级（必须先做，预计3人天）
#### 1. 补充测试安全网
- **任务**：编写核心流程集成测试用例，覆盖采集、评估、汇总、决策、报告全流程
- **收益**：避免重构过程中引入功能回归，提供自动化验证能力
- **实施方案**：
  - 使用pytest搭建测试框架
  - 编写mock数据，模拟各模块输入输出
  - 实现端到端测试，验证重构前后输出一致性
#### 2. 定义核心数据模型
- **任务**：使用pydantic定义所有核心数据结构（NewsItem、ExpertResult、AggregatedResult、TradingDecision）
- **收益**：解决字典传递的类型安全问题，减少字段缺失和类型错误
- **实施方案**：
  - 在`src/types/`目录下定义所有数据模型
  - 逐步替换现有模块中的字典传递
  - 增加数据校验逻辑，确保数据正确性
#### 3. 抽取公共HTTP客户端组件
- **任务**：实现通用异步HTTP客户端，封装重试、超时、日志、错误处理逻辑
- **收益**：解决4个采集器重复实现相同HTTP逻辑的问题，代码复用率提升70%
- **实施方案**：
  - 基于aiohttp封装通用客户端
  - 支持配置重试次数、超时时间、代理设置
  - 统一错误处理和日志格式
### 🟠 中优先级（重点做，预计8人天）
#### 1. 采集器模块重构
- **任务**：抽象BaseCollector基类，实现采集器自动注册发现机制
- **收益**：新增数据源无需修改主类代码，开发效率提升80%
- **实施方案**：
  ```python
  # 抽象基类
  class BaseCollector(ABC):
      @abstractmethod
      async def collect(self) -> List[NewsItem]:
          pass
  # 自动注册
  collector_registry = {}
  def register_collector(name: str):
      def decorator(cls):
          collector_registry[name] = cls
          return cls
      return decorator
  ```
  - 现有4个采集器继承BaseCollector
  - 主类中通过配置动态加载启用的采集器
#### 2. 引入依赖注入容器
- **任务**：使用dependency-injector框架实现依赖注入，解耦主类与各模块的直接依赖
- **收益**：主类代码量减少60%，模块可单独测试和替换
- **实施方案**：
  - 定义容器类，管理所有模块的生命周期
  - 模块依赖通过构造函数注入，不再直接实例化
  - 单元测试可轻松mock依赖，测试效率大幅提升
#### 3. 并行化改造
- **任务**：将串行执行的采集和评估流程改为并行执行
- **收益**：单次分析耗时降低至少40%，系统吞吐量提升
- **实施方案**：
  - 资讯采集：使用asyncio.gather并行执行所有启用的采集器
  - 专家评估：使用asyncio.Semaphore控制并发量，批量处理多条资讯
  - 结果汇总和决策：支持批量处理接口，减少循环调用开销
#### 4. 统一异常体系
- **任务**：定义标准化异常类和错误码，完善异常上下文信息
- **收益**：错误定位效率提升100%，可实现局部熔断降级
- **实施方案**：
  - 定义BaseException基类，包含错误码、上下文信息、堆栈跟踪
  - 各模块抛出标准化异常，不再直接抛出Exception
  - 实现全局异常处理器，统一日志记录和错误处理
### 🟢 低优先级（逐步做，预计10人天）
#### 1. 配置契约化改造
- **任务**：每个模块定义自己的配置schema，只接收需要的配置字段
- **收益**：配置变更影响范围可控，减少配置错误导致的线上问题
- **实施方案**：
  - 使用pydantic定义各模块的配置模型
  - 配置加载时自动验证，不合法配置提前报错
  - 模块只接收自己的配置对象，不再传递完整config字典
#### 2. AI调用网关实现
- **任务**：统一AI接口调用逻辑，实现多模型支持、流量控制、成本管控
- **收益**：AI调用逻辑集中管理，支持动态切换模型，成本可监控
- **实施方案**：
  - 抽象AIClient接口，支持OpenAI、Anthropic等多种实现
  - 实现调用统计、限流、降级、重试逻辑
  - 增加成本核算，记录每个模型的调用费用
#### 3. 流程编排改造
- **任务**：使用工作流引擎替代硬编码的主流程
- **收益**：支持动态调整分析步骤，无需修改核心代码
- **实施方案**：
  - 定义分析流程的各个节点和依赖关系
  - 使用工作流引擎编排执行顺序
  - 支持自定义流程，满足不同场景的分析需求
#### 4. 增量分析支持
- **任务**：实现数据增量标记，避免重复处理相同资讯
- **收益**：重复分析耗时降低80%，减少AI调用成本
- **实施方案**：
  - 基于URL和内容哈希实现去重
  - 记录已处理资讯的标记，增量分析时跳过
  - 支持缓存中间结果，减少重复计算
## 三、分模块具体重构建议
### 1. 主程序模块 (`src/main.py`)
#### 现有问题
- 过度耦合，直接实例化所有模块
- 包含不属于主类的去重逻辑
- 流程硬编码，扩展性差
#### 重构目标
```python
# 重构后主类
class StockSentimentAnalyzer:
    def __init__(self, container: Container):
        self.collector_manager = container.get(CollectorManager)
        self.expert_panel = container.get(ExpertPanel)
        self.aggregator = container.get(ResultAggregator)
        self.trader = container.get(ShortTermTrader)
        self.reporter = container.get(ReportGenerator)
    async def run_once(self):
        # 简洁的主流程
        news_list = await self.collector_manager.collect_all()
        if not news_list:
            return None
        expert_results = await self.expert_panel.evaluate_batch(news_list)
        aggregated_results = await self.aggregator.aggregate_batch(news_list, expert_results)
        trading_decisions = await self.trader.decide_batch(news_list, aggregated_results)
        return await self.reporter.generate(news_list, expert_results, aggregated_results, trading_decisions)
```
### 2. 采集器模块 (`src/collectors/`)
#### 现有问题
- 重复代码严重，复用率低
- 接口不统一，错误处理不一致
- 新增数据源需要修改主类
#### 重构目标
- 统一基类，复用公共逻辑
- 自动注册，动态加载
- 统一返回格式和错误处理
### 3. 专家模块 (`src/experts/`)
#### 现有问题
- 提示词硬编码在代码中
- 5个专家逻辑耦合在同一个类
- AI调用逻辑分散
#### 重构目标
- 提示词外部化，支持版本管理
- 每个专家独立实现，可动态配置
- AI调用通过统一网关
### 4. 数据模块 (`src/data/`)
#### 现有问题
- 接口不统一，上层需要适配
- 缓存逻辑耦合在实现中
- 没有数据校验层
#### 重构目标
- 统一DataProvider接口
- 缓存作为独立中间件
- 增加数据清洗和校验层
## 四、风险控制措施
### 1. 功能一致性保障
- 重构前录制现有系统的输入输出样本
- 每完成一个重构点，运行对比测试，确保输出一致
- 上线前进行灰度验证，对比重构前后的结果差异
### 2. 回滚方案
- 每个重构点保持向下兼容，新旧实现可并存
- 提供配置开关，可快速切换回旧实现
- 小步提交，每个commit对应一个独立的重构点，可单独回滚
### 3. 性能监控
- 重构前后采集核心性能指标（分析耗时、采集成功率、AI调用耗时等）
- 上线后持续监控性能变化，及时发现性能 regression
### 4. 代码质量控制
- 重构代码必须通过单元测试和集成测试
- 代码覆盖率不低于重构前水平
- 代码评审重点关注兼容性和正确性
